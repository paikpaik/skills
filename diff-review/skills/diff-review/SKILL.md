# Diff Review — 커밋 전 코드 리뷰

## 사용법

```
# 현재 모든 변경사항 (staged + unstaged)
/diff-review

# staged만 (git add된 것)
/diff-review --staged

# 특정 커밋 범위
/diff-review HEAD~3
/diff-review main..HEAD

# 특정 파일만
/diff-review src/controllers/contents/reward/
```

---

## Phase 0: 범위 결정

인수에 따라 diff 범위를 결정한다.

| 인수 | 명령어 |
|------|--------|
| 없음 | `git diff HEAD` (staged + unstaged 전체) |
| `--staged` | `git diff --staged` |
| `HEAD~N` / `ref..ref` | `git diff {ref}` |
| 경로 | `git diff HEAD -- {path}` |

```bash
# 변경된 파일 목록 먼저 확인
git diff HEAD --name-status

# 통계 확인
git diff HEAD --stat
```

변경 파일이 없으면 중단하고 안내한다.

## Phase 1: 변경사항 파악

### 1-1. 변경 파일 분류

변경된 파일을 역할별로 분류한다:

| 분류 | 패턴 예시 |
|------|-----------|
| 비즈니스 로직 | `controllers/`, `services/`, `handlers/` |
| 라우트 | `routes/`, `router/` |
| 모델 / DB | `models/`, `migrations/`, `schemas/` |
| 설정 | `config/`, `*.config.*`, `.env*` |
| 타입 | `*.d.ts`, `types/`, `interfaces/` |
| 테스트 | `*.spec.*`, `*.test.*`, `__tests__/` |
| 인프라 | `Dockerfile`, `*.yml`, `*.yaml`, `*.sh` |
| 기타 | 그 외 |

### 1-2. diff 읽기

파일별로 diff를 읽는다. 파일이 크거나 변경량이 많으면 핵심 변경 부분(함수 단위)에 집중한다:

```bash
git diff HEAD -- {file}
```

변경량이 많은 경우 (`+` 라인 500줄 이상) 파일을 나눠 순차적으로 읽는다.

### 1-3. 관련 파일 추적

변경된 코드가 참조하는 외부 모듈, 공통 함수, 타입을 파악해 **변경의 영향 범위**를 추정한다:

```bash
# 변경된 함수/클래스를 다른 파일에서 import하는지 확인
grep -r "{changed_symbol}" src/ --include="*.ts" -l
```

## Phase 2: 리뷰 분석

아래 관점으로 변경사항을 분석한다.

### 2-1. 버그 / 잠재적 오류 [BLOCK]

커밋하면 실제로 문제가 생길 가능성이 있는 항목:

- 조건 분기 누락 (null/undefined 미처리, 엣지 케이스)
- async/await 빠진 비동기 호출
- 에러 핸들링 없는 외부 호출 (DB, Redis, 외부 API)
- 변수 참조 오류, 잘못된 타입 사용
- 트랜잭션 미처리 (여러 DB 작업이 원자적으로 묶여야 하는 경우)
- 무한 루프 / 재귀 탈출 조건 누락

### 2-2. 누락된 것들 [WARN]

작성하려 했지만 빠트렸을 가능성이 있는 항목:

**코드 연동**
- 새 함수/모듈을 만들었는데 export하지 않음
- 라우트 등록 누락 (controller 만들었는데 router에 연결 안 함)
- 에러 코드 추가했는데 문서나 enum 정의에 반영 안 됨

**DB / 마이그레이션**
- 모델에 컬럼 추가했는데 migration 파일 없음
- 인덱스 추가/삭제 누락
- 기존 데이터 호환성 고려 (컬럼 NOT NULL 추가 시 기존 행 처리)

**설정 / 환경변수**
- 코드에서 새 `process.env.XXX` 사용했는데 `.env.example` 미반영
- config 파일에 새 키 추가했는데 환경별(dev/stage/prod) 누락

**테스트**
- 변경된 함수에 대한 테스트 없음 (테스트 파일이 프로젝트에 존재하는 경우)

**타입**
- JS에서 새 필드 사용했는데 타입 정의 미업데이트
- 인터페이스/DTO 변경이 사용처에 미반영

### 2-3. 고려해야 할 것들 [SUGGEST]

틀린 건 아니지만 생각해볼 가치가 있는 항목:

- **성능**: N+1 쿼리 가능성, 루프 안 DB/Redis 호출, 캐싱 대상 여부
- **동시성**: 분산 락 없는 중복 요청 처리, race condition 가능성
- **보안**: 입력값 검증 범위, 권한 체크 위치, 민감 데이터 로깅
- **하위 호환성**: API 응답 구조 변경 시 클라이언트 영향, 필드 제거/이름 변경
- **부작용**: 공통 함수 수정이 다른 호출처에 미치는 영향
- **롤백 가능성**: 이 변경을 롤백할 때 데이터 정합성이 깨지지 않는지

## 분석 품질 원칙

- BLOCK은 실제로 문제가 생길 가능성이 높은 것만 올린다. 스타일이나 취향 차이는 BLOCK이 아니다.
- WARN은 "이거 빠트린 거 맞죠?"처럼 확인이 필요한 것. 틀렸다고 단정하지 않는다.
- SUGGEST는 트레이드오프가 있는 것. 반드시 수정해야 한다고 강요하지 않는다.
- 변경 파일이 많을 때(10개 이상)는 비즈니스 로직 파일을 우선 분석하고, 설정/인프라 파일은 간략히 처리한다.
- 코드를 읽다가 관련 파일을 추가로 확인해야 할 때 적극적으로 읽는다. 변경 파일만 보면 놓치는 게 많다.
